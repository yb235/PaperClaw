## Paper Metadata (metadata.json)

**Location:** `papers/{paper_title}/metadata.json`

**Created by:** Daily search skill (initial) → Paper review skill (updated with scores)

**Read by:** Weekly report skill

This is the most important per-paper file. It starts with basic info from the search and gets enriched with scores after evaluation.

### Full Schema

```json
{
  "arxiv_id": "1910.03193",
  "title": "Learning Nonlinear Operators via DeepONet Based on the Universal Approximation Theorem of Operators",
  "short_title": "DeepONet",
  "authors": ["Lu Lu", "Pengzhan Jin", "George Em Karniadakis"],
  "venue": "Nature Machine Intelligence",
  "publication_date": "2019-10-08",
  "citations": 3144,
  "evaluated_date": "2026-03-01",
  "scores": {
    "engineering_value": 9.0,
    "architecture_innovation": 10.0,
    "theoretical_contribution": 9.0,
    "result_reliability": 9.0,
    "impact": 10.0,
    "final_score": 9.33
  },
  "keywords": ["operator learning", "DeepONet", "neural operators"],
  "relevance": "very_high",
  "summary": "Brief one-line summary of the paper",
  "last_updated": "2026-03-01T11:47:43.560831"
}
```

### Field-by-Field Explanation

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `arxiv_id` | string | Yes | The arXiv identifier (e.g., `"1910.03193"` or `"2401.12345v2"`). This is the primary key used for deduplication. |
| `title` | string | Yes | Full paper title as it appears on arXiv. |
| `short_title` | string | Yes | A shortened, filesystem-safe version of the title used as the folder name. Generated by removing common words and truncating. |
| `authors` | string[] | Yes | List of author names in order of appearance on the paper. |
| `venue` | string | No | Publication venue (e.g., `"NeurIPS 2024"`, `"Nature Machine Intelligence"`). May be empty for preprints. Sourced from Semantic Scholar. |
| `publication_date` | string | Yes | ISO 8601 date when the paper was first published (e.g., `"2019-10-08"`). Used for age calculation in Date-Citation weighting. |
| `citations` | integer | No | Total citation count from Semantic Scholar. Updated when paper is evaluated. |
| `evaluated_date` | string | Yes | ISO 8601 date/datetime when the paper was evaluated by PaperClaw. |
| `scores` | object | No | Present only after evaluation. Contains all scoring dimensions. |
| `scores.engineering_value` | float | — | Score 1-10: How useful is this paper for real engineering applications? |
| `scores.architecture_innovation` | float | — | Score 1-10: How novel is the network/method design? |
| `scores.theoretical_contribution` | float | — | Score 1-10: Does it advance mathematical understanding? |
| `scores.result_reliability` | float | — | Score 1-10: Are the experiments rigorous and reproducible? |
| `scores.impact` | float | — | Score 1-10: Combined impact score after Date-Citation adjustment. |
| `scores.final_score` | float | — | The weighted final score: `(4-dim avg × 0.9) + (impact × 0.1)`. |
| `keywords` | string[] | No | Relevant keywords extracted from the paper. |
| `relevance` | string | No | Relevance tier: `"very_high"`, `"high"`, `"medium"`, `"low"`. |
| `summary` | string | No | Brief one-line summary (the full summary lives in `summary.md`). |
| `last_updated` | string | Yes | ISO 8601 datetime of the last modification to this file. |

## Evaluated Papers Registry (evaluated_papers.json)

**Location:** `papers/evaluated_papers.json`

**Created by:** Paper review skill (via `update_registry.py`)

**Read by:** Daily search skill (for dedup), weekly report skill (for report generation)

This is the **central registry** — the single source of truth for all papers that have been evaluated.

### Full Schema

```json
{
  "papers": [
    {
      "arxiv_id": "1910.03193",
      "title": "Learning Nonlinear Operators via DeepONet...",
      "short_title": "DeepONet",
      "scores": {
        "final_score": 9.33
      },
      "evaluated_date": "2026-03-01T11:47:43Z"
    },
    {
      "arxiv_id": "2010.08895",
      "title": "Fourier Neural Operator for Parametric PDEs",
      "short_title": "FNO",
      "scores": {
        "final_score": 8.92
      },
      "evaluated_date": "2026-03-02T14:22:00Z"
    }
  ],
  "last_updated": "2026-03-02T14:22:00Z"
}
```

### Field-by-Field Explanation

| Field | Type | Description |
|-------|------|-------------|
| `papers` | array | Array of paper entries, one per evaluated paper. |
| `papers[].arxiv_id` | string | arXiv ID — used for deduplication lookups. |
| `papers[].title` | string | Full paper title — used for title-based deduplication. |
| `papers[].short_title` | string | Folder name for this paper's data. |
| `papers[].scores` | object | Contains at minimum `final_score`. |
| `papers[].scores.final_score` | float | The final weighted score used for ranking. |
| `papers[].evaluated_date` | string | ISO 8601 datetime of when this paper was evaluated. Used by weekly report to filter recent papers. |
| `last_updated` | string | ISO 8601 datetime of when the registry was last modified. |

> **Rationale:** The registry intentionally stores minimal data per paper (just enough for dedup and ranking). Full details live in each paper's `metadata.json`. This keeps the registry file small and fast to read, even with hundreds of papers.

## Search Log (search_log.json)

**Location:** `search_logs/YYYY-MM-DD_search_log.json`

**Created by:** Daily search skill

**Read by:** Humans (for debugging and auditing)

### Full Schema

```json
{
  "search_date": "2026-03-01T20:00:00Z",
  "total_searched": 270,
  "after_dedup": 245,
  "skipped_evaluated": 200,
  "selected_count": 3,
  "selected_papers": [
    {
      "arxiv_id": "2401.12345",
      "title": "A New Mesh-Aware Neural Operator",
      "relevance_score": 95
    }
  ],
  "skipped_details": [
    {
      "arxiv_id": "2312.54321",
      "reason": "ID已评估"
    },
    {
      "arxiv_id": "2401.99999",
      "reason": "标题已评估"
    }
  ]
}
```

### Field-by-Field Explanation

| Field | Type | Description |
|-------|------|-------------|
| `search_date` | string | When this search was performed. |
| `total_searched` | integer | Total number of papers returned by all 9 arXiv queries combined. |
| `after_dedup` | integer | Papers remaining after internal deduplication. |
| `skipped_evaluated` | integer | Papers skipped because they were already in the registry. |
| `selected_count` | integer | Number of papers selected for evaluation (typically 3). |
| `selected_papers` | array | Details of each selected paper. |
| `selected_papers[].relevance_score` | integer | The relevance score from keyword matching. |
| `skipped_details` | array | Why each skipped paper was skipped — useful for debugging. |
| `skipped_details[].reason` | string | `"ID已评估"` (ID already evaluated) or `"标题已评估"` (title already evaluated). |

> **Rationale:** Search logs exist purely for debugging and auditing. If the agent starts missing important papers, you can check the logs to see if they were filtered out and why.

## Evaluation Task List (pending_evaluation.json)

**Location:** `pending_evaluation_YYYY-MM-DD.json`

**Created by:** Daily search skill

**Read by:** The agent (to know which papers to evaluate next)

### Full Schema

```json
{
  "date": "2026-03-01",
  "tasks": [
    {
      "priority": 1,
      "arxiv_id": "2401.12345",
      "title": "A New Mesh-Aware Neural Operator",
      "short_title": "MeshAwareNeuralOp",
      "paper_dir": "/home/gem/.openclaw/workspace/3d_surrogate_proj/papers/MeshAwareNeuralOp",
      "relevance_score": 95,
      "status": "pending"
    },
    {
      "priority": 2,
      "arxiv_id": "2401.67890",
      "title": "Geometry-Informed DeepONet",
      "short_title": "GeoDeepONet",
      "paper_dir": "/home/gem/.openclaw/workspace/3d_surrogate_proj/papers/GeoDeepONet",
      "relevance_score": 87,
      "status": "pending"
    }
  ]
}
```

### Field-by-Field Explanation

| Field | Type | Description |
|-------|------|-------------|
| `date` | string | The date this task list was generated. |
| `tasks` | array | Ordered list of papers to evaluate. |
| `tasks[].priority` | integer | 1 = highest priority. Based on relevance score. |
| `tasks[].arxiv_id` | string | arXiv ID of the paper. |
| `tasks[].title` | string | Full paper title. |
| `tasks[].short_title` | string | Folder name where the paper's files are stored. |
| `tasks[].paper_dir` | string | Absolute path to the paper's directory. |
| `tasks[].relevance_score` | integer | How relevant this paper is to the research domain. |
| `tasks[].status` | string | `"pending"`, `"in_progress"`, or `"completed"`. |

> **Rationale:** The task list acts as a work queue. The "deferred evaluation" pattern (search first, evaluate later) allows the agent to prioritize papers and potentially even let a human review the list before evaluation begins.

## Paper Summary (summary.md)

**Location:** `papers/{paper_title}/summary.md`

**Created by:** The LLM during paper review

**Read by:** Humans, weekly report skill

### Template Structure

```markdown
# {Paper Title}

## 论文基本信息
- **arXiv ID**: {id}
- **作者**: {authors}
- **发表日期**: {date}
- **引用数**: {citations}

---

## 1. 这篇论文解决了什么问题？
{Answer — typically 2-3 paragraphs}

## 2. 这是一个新问题吗？
{Answer}

## 3. 这篇论文的科学假设是什么？
{Answer}

## 4. 相关研究和关键人物有哪些？
{Answer}

## 5. 解决方案的关键是什么？
{Answer}

## 6. 实验是如何设计的？
{Answer}

## 7. 使用了哪些数据集？是否开源？
{Answer}

## 8. 实验结果是否支持假设？
{Answer}

## 9. 论文的主要贡献是什么？
{Answer}

## 10. 未来工作和建议？
{Answer}
```

> **Rationale:** Using a fixed set of 10 questions ensures every summary covers the same dimensions, making papers directly comparable. The questions follow a research paper reading methodology framework.

## Paper Score Report (scores.md)

**Location:** `papers/{paper_title}/scores.md`

**Created by:** The LLM during paper review

**Read by:** Humans, weekly report skill

### Template Structure

```markdown
# {Paper Title} - 评分报告

## 基本信息
- **标题**: {title}
- **arXiv ID**: {id}
- **发表日期**: {date}
- **引用数**: {citations}

---

## 四维评分

### 1. 工程应用价值 (Engineering Application Value)
**评分**: {score}/10

**评分理由**:
{Detailed reasoning — typically 3-5 sentences}

### 2. 网络架构创新 (Architecture Innovation)
**评分**: {score}/10

**评分理由**:
{Detailed reasoning}

### 3. 理论贡献 (Theoretical Contribution)
**评分**: {score}/10

**评分理由**:
{Detailed reasoning}

### 4. 结果可靠性 (Result Reliability)
**评分**: {score}/10

**评分理由**:
{Detailed reasoning}

---

## 影响力评分 (Impact Score)

**基础影响力**: {base_score}/10
**论文年龄**: {age_months} 个月
**引用数**: {citations}
**引用密度**: {density} 次/月
**调整因子**: +{adjustment}
**最终影响力评分**: {final_impact}/10

---

## 综合评分

**四维基础评分**: {four_dim_score}/10
**最终综合评分**: {final_score}/10

**计算公式**: {four_dim_score} × 0.9 + {impact_score} × 0.1 = {final_score}

---

## 总结与建议
{Overall assessment and recommendations}
```

## Model Configuration (models.json)

**Location:** `agent/models.json`

### Full Schema

```json
{
  "providers": {
    "{provider_name}": {
      "baseUrl": "http://example.com/v1",
      "apiKey": "YOUR_API_KEY_HERE",
      "api": "openai-completions",
      "models": [
        {
          "id": "{model-id}",
          "name": "{Display Name}",
          "contextWindow": 200000,
          "maxTokens": 8192
        }
      ]
    }
  }
}
```

### Field-by-Field Explanation

| Field | Type | Description |
|-------|------|-------------|
| `providers` | object | Map of provider names to their configurations. |
| `baseUrl` | string | The LLM API endpoint URL. |
| `apiKey` | string | API authentication key. |
| `api` | string | API protocol type. `"openai-completions"` means it uses the OpenAI chat completions format. |
| `models[].id` | string | Model identifier sent in API requests. |
| `models[].name` | string | Human-friendly display name. |
| `models[].contextWindow` | integer | Maximum input tokens the model can handle. 200K allows reading full papers. |
| `models[].maxTokens` | integer | Maximum output tokens per response. 8192 allows detailed summaries. |

## Schedule Configuration (schedules.json)

**Location:** `agent/schedules.json`

### Full Schema

```json
{
  "schedules": [
    {
      "name": "{Schedule Name}",
      "schedule": {
        "kind": "cron",
        "expr": "{cron expression}",
        "tz": "{timezone}"
      },
      "payload": {
        "kind": "agentTurn",
        "message": "{message to send to agent}"
      },
      "sessionTarget": "isolated"
    }
  ]
}
```

### Field-by-Field Explanation

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Human-readable name for this schedule. |
| `schedule.kind` | string | Schedule type. Currently only `"cron"` is used. |
| `schedule.expr` | string | Standard cron expression (minute hour day-of-month month day-of-week). |
| `schedule.tz` | string | IANA timezone name. All PaperClaw schedules use `"Asia/Singapore"`. |
| `payload.kind` | string | `"agentTurn"` means the schedule sends a message to start an agent conversation. |
| `payload.message` | string | The actual text message sent to the agent. Written in Chinese to match the agent's default language. |
| `sessionTarget` | string | `"isolated"` = each run gets its own session. Prevents state leakage between runs. |

## Semantic Scholar API Response

When PaperClaw fetches data from Semantic Scholar, here's what a response looks like:

```json
{
  "paperId": "abc123def456",
  "title": "Learning Nonlinear Operators via DeepONet...",
  "authors": [
    {
      "authorId": "12345",
      "name": "Lu Lu"
    }
  ],
  "year": 2019,
  "publicationDate": "2019-10-08",
  "citationCount": 3144,
  "influentialCitationCount": 75,
  "venue": "Nature Machine Intelligence",
  "openAccessPdf": {
    "url": "https://arxiv.org/pdf/1910.03193"
  },
  "isOpenAccess": true
}
```

> **Key fields used:** `citationCount` (for scoring), `publicationDate` (for age calculation), `venue` (for metadata), `influentialCitationCount` (for quality assessment).

## arXiv API Response

The arXiv API returns XML. After parsing, each paper entry looks like:

```json
{
  "id": "http://arxiv.org/abs/1910.03193v5",
  "arxiv_id": "1910.03193",
  "title": "Learning Nonlinear Operators via DeepONet...",
  "summary": "The development of efficient surrogates...",
  "authors": ["Lu Lu", "Pengzhan Jin", "George Em Karniadakis"],
  "published": "2019-10-08T17:59:59Z",
  "updated": "2021-03-18T15:22:00Z",
  "categories": ["cs.LG", "physics.comp-ph"],
  "pdf_url": "http://arxiv.org/pdf/1910.03193v5"
}
```

> **Key fields used:** `arxiv_id` (primary key), `title` and `summary` (for relevance scoring), `pdf_url` (for downloading), `published` (for age calculation).

## Design Rationale

### Why JSON for structured data?

- **Machine-readable**: Programs can parse it without regex
- **Well-understood**: Every programming language has JSON support
- **Self-documenting**: Field names describe the data
- **Flexible**: Easy to add new fields without breaking existing readers

### Why Markdown for human-readable documents?

- **Readable as-is**: You don't need a special viewer
- **Renderable**: GitHub, knowledge bases, and chat tools render it beautifully
- **Writable by LLMs**: Language models produce high-quality Markdown naturally
- **Diffable**: Changes show up clearly in version control

### Why both metadata.json AND scores.md for the same paper?

- `metadata.json` is for **machines** — the weekly report script reads scores from here
- `scores.md` is for **humans** — researchers read the detailed reasoning here
- Early versions only had `scores.md` and extracted scores with regex, which broke frequently
- The dual-file approach is more reliable and serves both audiences

### Why a flat registry instead of a database?

- **Simplicity**: No database setup, no migration scripts, no connection pooling
- **Portability**: Copy the workspace folder and everything works
- **Debuggability**: Open the file in a text editor to see all data
- **Sufficient scale**: With ~3 papers/day, even after a year there are only ~1,000 entries — trivially handled by JSON
